# Type Propagation Flow

Types flow strictly bottom-up across six packages. No build step — all packages export `.ts` source directly (Wrangler bundles at deploy).

## Package Chain

```
@pignal/db                    Schema + types (no logic)
  |
  v
@pignal/core                  Business logic + routes + MCP + validation + federation
  |
  v
@pignal/render                Shared rendering components, lib, i18n, static assets, styles
  |
  v
@pignal/templates             Self-contained template folders (config + JSX), catalog, seed data
  |
  v
@pignal/web                   Admin dashboard + routing only
  |
  v
@pignal/server                Deployment wiring + auth
```

## Key Type Locations

### `@pignal/db` — Source of Truth

- `db/src/schema.ts` — Drizzle table definitions (`items`, `itemTypes`, `typeActions`, `workspaces`, `settings`, `apiKeys`). Drizzle infers `ItemSelect`, `ItemInsert`, etc.
- `db/src/types.ts` — Hand-written interfaces:
  - `ItemStoreRpc` — 57-method interface defining all business operations
  - `ItemWithMeta` — Item joined with type name, workspace name, validation action label, parsed tags array
  - `ListParams`, `CreateParams`, `UpdateParams`, `VouchParams` — operation parameter types
  - `MetadataResult` — combined types + workspaces + settings

### `@pignal/core` — Implements and Consumes

- `core/src/store/item-store.ts` — `class ItemStore implements ItemStoreRpc` (accepts any Drizzle SQLite DB)
- `core/src/routes/*.ts` — Route factories accept `RouteFactoryConfig { getStore, middleware? }`, call `config.getStore(c)` to get `ItemStoreRpc`
- `core/src/validation/schemas.ts` — Zod schemas (independent of Drizzle types, but field names must match)
- `core/src/mcp/tools.ts` — Tool functions accept `ItemStoreRpc` + validated params
- `core/src/federation/types.ts` — `WellKnownResponse`, `WellKnownTemplate` — federation discovery types
- `core/src/types.ts` — `RouteFactoryConfig`, `PUBLIC_ITEM_FIELDS`

### `@pignal/render` — Shared Rendering Layer

- `render/src/components/*.tsx` — Shared JSX components (layout, public-layout, pagination, type-badge, empty-state, item-feed, type-sidebar, source-action-bar, json-ld, icons, cta-block, language-switcher, action-form, item-card, visibility-badge, testimonials)
- `render/src/lib/*.ts` — Shared utilities (theme, seo, markdown, time, css-sanitize, static-versions, geo, rss, directives)
- `render/src/i18n/` — i18n setup, locale files, translation utilities
- `render/src/static/` — Static assets (tailwind.css, app.js, logo, htmx)
- `render/src/styles/input.css` — Design tokens (`@theme`), base styles, `@layer components`
- `render/src/types.ts` — Shared rendering type definitions

### `@pignal/templates` — Self-Contained Template Folders

- `templates/src/<name>/config.ts` — Per-template `TemplateConfig` (profile + vocabulary + SEO + MCP with schemaDescriptions)
- `templates/src/<name>/index.tsx` — Exports `Template` object with JSX components + `profile: config.profile`
- `templates/src/<name>/source-page.tsx` — Source page JSX component
- `templates/src/<name>/item-post.tsx` — Item post JSX component
- `templates/src/<name>/layout.tsx` — Template layout component
- `templates/src/catalog.ts` — `TEMPLATE_CATALOG` — static registry of all templates (shipped, planned, rejected) with `CatalogEntry` type
- `templates/src/all-configs.ts` — Barrel file exporting all template configs (used by hub for multi-template lookups)
- `templates/src/config.ts` — Shared `TemplateConfig` type definitions and classification enums
- `templates/src/types.ts` — `Template` interface (JSX components + `profile: TemplateProfile`), prop types (`SourcePageProps`, `ItemPostProps`, etc.)
- `templates/src/registry.ts` — `getTemplate()` returns the build-time resolved template from `_resolved.ts`
- `templates/src/_resolved.ts` — Auto-generated by `resolve-template` script, re-exports the active template
- `templates/src/index.ts` — Barrel exports for all types and catalog

### `@pignal/web` — Admin Dashboard Only

- `web/src/components/*.tsx` — Admin-only components (app-layout, create-section, feed-item, filter-sidebar, flash, form-dropdown, managed-list, page-header, skeleton, stat-card, status-badge)
- `web/src/pages/*.tsx` — Admin dashboard pages
- `web/src/middleware/` — Session, CSRF, security headers, analytics
- `web/src/lib/` — Admin-only utilities (cookie.ts, htmx.ts, slug.ts)

### `@pignal/server` — Wires Everything

- `server/src/middleware/store.ts` — Creates `ItemStore` from `env.DB` (D1), sets on Hono context as `store`
- `server/src/index.ts` — Passes `{ getStore: (c) => c.get('store') }` to route factories. Imports `resolvedConfig` from `@pignal/templates/resolved`
- `server/src/mcp/agent.ts` — Gets store from context, passes to core MCP tool functions

## When You Change a Type

1. **Add/modify column** -> Update `db/src/schema.ts` + `db/src/types.ts`
2. **Run `pnpm type-check`** -> TypeScript shows every downstream breakage in core/render/templates/web/server
3. **Fix core** -> Update store methods, validation schemas, route handlers
4. **Fix render** -> Update shared components if they display/use the changed type
5. **Fix templates** -> Update template JSX components if needed
6. **Fix web** -> Update admin pages/components if needed
7. **Fix server** -> Update middleware, agent as needed
8. **Write migration SQL** -> `server/migrations/NNNN_*.sql`
